# 新网页从创意到上线的标准工作流

这份文档用于以后制作新的本地网页，并把它发布成可以分享的公网链接。目标是让流程可重复、可检查、可交给 AI 或另一个人继续执行。

默认路线：先做一个能传播的最小版本，再用 GitHub 管理代码，最后通过 Netlify 静态托管上线。

## 1. 创意定义

开始写代码前，先把网页的最小目标说清楚。

必须回答：

- 一句话目标：这个网页帮用户完成什么？
- 目标用户：谁最可能愿意点开？
- 使用场景：用户会在什么情况下看到、打开、使用它？
- 传播点：用户为什么愿意截图、转发或发给朋友？
- 最小可上线版本：去掉所有非必要功能后，第一版必须保留什么？

建议模板：

```md
网页名称：
一句话目标：
目标用户：
典型使用场景：
用户完成后得到什么：
为什么值得分享：
第一版必须有：
第一版暂时不做：
```

决策检查点：

- 这个网页为什么值得别人点开？
- 用户 30 秒内能不能理解它是干什么的？
- 用户完成后有没有一个明确结果、反馈或下一步行动？
- 这个网页能不能先做成纯前端静态页面？
- 有没有必须隐藏的密钥、登录、数据库或后端逻辑？如果没有，优先静态托管。

## 2. 本地网页制作

先做可用体验，再整理技术结构。不要一开始追求复杂后台、账号系统或过度抽象。

推荐顺序：

1. 先完成主体验：入口、核心交互、结果页或完成状态。
2. 再补齐移动端布局、按钮状态、错误状态和分享文案。
3. 如果只是轻量网页，可以先用静态 HTML/CSS/JS 验证创意。
4. 如果需要长期维护，整理为 Next.js + TypeScript 项目。
5. 把数据、文案、计算逻辑和页面表现分开，方便以后迭代。

当前项目的推荐本地命令：

```bash
npm install
npm run dev
```

当前项目的核心结构：

- `app/page.tsx`：主页面和用户交互。
- `app/globals.css`：视觉样式和响应式布局。
- `lib/`：题目数据、结果计算、类型定义等可复用逻辑。
- `public/`：静态资源和 Netlify 响应头配置。
- `next.config.ts`：静态导出配置。

如果是新网页，可以沿用这个结构，但先保证体验跑通，再拆模块。

## 3. 本地验证

每次准备提交或上线前，都要先本地检查。

基础检查：

```bash
npm run build
```

手动体验清单：

- 首页能正常打开。
- 核心流程能从开始走到结束。
- 刷新页面后没有明显报错。
- 桌面端布局不挤、不重叠。
- 手机端文字不溢出，按钮容易点击。
- 分享按钮、复制文案或结果展示符合预期。
- 页面标题、说明文字和最终结果没有临时占位内容。

如果网页主要靠传播，一定要额外检查：

- 首屏是否能让人立刻知道要不要继续。
- 结果页是否适合截图。
- 分享出去的文案是否自然、简短、能引发点击。

## 4. Git 管理

Git 用来保存版本、记录决策，并和 GitHub、Netlify 串起来。

每次开始改动前：

```bash
git status
```

推荐提交节奏：

- 一个 commit 只做一个明确主题。
- 创意文案、页面结构、视觉样式、部署配置尽量分开提交。
- 如果某次代码修改改变了产品方向、部署方式或使用流程，要同步更新文档。
- 不要把大量无关改动混进一次提交。

常用命令：

```bash
git add .
git commit -m "Describe the iteration"
git push
```

提交信息建议写清楚做了什么，例如：

```bash
git commit -m "Add Netlify deployment config"
git commit -m "Refine quiz result copy"
git commit -m "Add reusable website workflow docs"
```

## 5. GitHub 联动

GitHub 在这个流程里承担三个作用：

- 代码备份：本地电脑出问题也不会丢。
- 版本记录：以后能看清每一步为什么改。
- 部署来源：Netlify 可以直接连接 GitHub 仓库自动部署。

推荐流程：

1. 本地完成一轮可运行修改。
2. 执行构建检查。
3. 提交到 Git。
4. 推送到 GitHub。
5. 让 Netlify 从 GitHub 仓库自动构建和发布。

如果之后让我继续做新网页，可以直接要求我：

```md
请先检查当前 Git 状态，再按主题提交，并把可上线版本推送到 GitHub。
```

## 6. Netlify 上线

优先方式：连接 GitHub 仓库自动部署。

Netlify 项目设置使用：

```txt
Build command: npm run build
Publish directory: out
Node version: 24.14.0
```

当前项目已经有 `netlify.toml`，关键配置是：

```toml
[build]
  command = "npm run build"
  publish = "out"

[build.environment]
  NODE_VERSION = "24.14.0"
```

这条路线适合长期维护：每次推送到 GitHub 后，Netlify 自动构建并更新公网链接。

备用方式：手动生成 `out` 文件夹后拖拽上传。

Windows 下可以双击：

```txt
build-share-folder.cmd
```

或者运行：

```bash
npm run build
```

构建完成后，把 `out` 文件夹上传到 Netlify Drop。这个方式适合临时分享、快速验证，缺点是后续每次更新都要重新上传。

## 7. 上线后检查

拿到公网链接后，不要直接发出去，先做一次真实访问检查。

上线检查清单：

- 公网链接能打开。
- 手机流量网络下能打开。
- 微信、聊天软件或社交平台里能正常打开。
- 首页、核心流程、结果页都能访问。
- 样式和本地预览一致。
- 分享文案、复制结果或截图效果正常。
- 页面没有明显加载失败、乱码或空白。

如果发现问题：

1. 先记录问题现象和链接。
2. 本地复现并修复。
3. 执行 `npm run build`。
4. 提交到 Git。
5. 推送到 GitHub。
6. 等 Netlify 重新部署。
7. 再检查公网链接。

## 8. 下一轮迭代

上线不是结束，第一版的重点是收集真实反馈。

建议记录：

- 用户是否愿意点开。
- 用户是否能快速理解。
- 用户是否能完成核心流程。
- 用户是否愿意分享。
- 用户在哪一步退出或卡住。
- 用户最常截图或转发哪个页面。
- 用户反馈最强烈的是文案、视觉、流程还是结果。

下一轮迭代时，按这个顺序处理：

1. 先修影响打开、完成、分享的关键问题。
2. 再优化文案和结果表达。
3. 再补充视觉细节和动效。
4. 最后才考虑后端、数据库、账号、埋点或复杂功能。

每一轮都重复：

```txt
定义改动目标 -> 本地修改 -> 本地验证 -> Git 提交 -> GitHub 推送 -> Netlify 部署 -> 公网检查 -> 记录反馈
```

## 给 AI 的复用提示词

以后要制作一个新网页时，可以直接使用这段提示词：

```md
请按 `WEB_WORKFLOW.md` 的流程，帮我从一个新创意开始制作网页、接入 GitHub、部署到 Netlify，并在每一步完成后更新文档和检查结果。

默认要求：
- 先确认网页的一句话目标、目标用户、使用场景、传播点和最小可上线版本。
- 优先做静态网页或 Next.js 静态导出，不要一开始引入后端。
- 本地必须能运行和构建。
- 用 Git 按主题提交，并推送到 GitHub。
- Netlify 使用 `npm run build` 和 `out` 目录部署。
- 上线后检查公网链接、手机访问和分享效果。
- 如果修改了产品方向、技术路线或部署方式，同步更新项目文档。
```

如果只是让 AI 继续当前项目，可以说：

```md
请先阅读 `WEB_WORKFLOW.md`、`README.md` 和 `PROJECT_BACKGROUND.md`，然后按当前项目的网页制作、GitHub、Netlify 流程继续迭代。
```
